第09章 上下文工程

1. 本章目标

上下文指模型采样时可见的全部 tokens，包括系统提示、用户输入、工具说明、对话历史、检索结果、外部文件、记忆、笔记和中间状态。

提示工程关注“怎么写提示词”。

上下文工程关注“哪些信息应该进入上下文，以及如何收集、筛选、组织、压缩和维护”。

随着智能体从单轮问答走向长时程任务，模型需要在多次推理循环中持续使用状态信息，仅靠静态提示词已经不够。

2. 为什么上下文工程重要

上下文窗口看似越来越大，但并不意味着可以无限塞信息。随着 token 数增加，模型会出现 上下文腐蚀：信息越多，模型越容易忽略关键内容。

Transformer 的注意力关系近似随长度形成 \(n^2\) 级别交互。上下文越长，注意力越分散，模型对关键信息的定位和长程依赖会下降。

因此，上下文应被视为有限资源。优秀的上下文工程目标是：

用尽可能少但高信号密度的 tokens，最大化模型产生期望行为的概率。

3. 有效上下文的组成

系统提示

系统提示应清晰、直接、层次适中。常见误区：

  • 过度硬编码复杂规则，导致维护困难。
  • 过于抽象，缺少具体行为约束。

建议用 Markdown 或 XML 分区组织角色、背景、任务、工具说明和输出格式。

工具

工具是智能体与外部世界交互的接口。工具设计应：

  • 职责单一。
  • 接口语义清晰。
  • 参数描述明确。
  • 错误处理友好。
  • 返回结果 token 友好。

工具越多不一定越好。边界模糊的工具集会增加模型选择成本。

示例

Few-shot 示例能显著改善模型行为，但不应堆砌边界条件。好的示例应典型、多样、直接刻画期望输出。

4. JIT 上下文与智能体式搜索

传统 RAG 常在推理前一次性检索信息，而上下文工程更强调 及时上下文（Just-in-time Context）。

JIT 上下文的思路是：

  • 不预先塞入所有信息。
  • 保留轻量引用，如文件路径、URL、数据库查询条件。
  • 让智能体运行时通过工具按需读取。
  • 逐步披露信息，避免上下文污染。

这与人类工作方式类似：我们不会记住所有资料，而是知道如何查找和利用资料。

5. 长时程任务中的上下文管理

长时程任务可能跨越大量工具调用、多轮对话和多阶段目标。仅依赖大上下文窗口并不可靠，需要三类机制。

压缩整合

当上下文接近上限时，对历史进行高保真总结，用摘要接续新上下文。重点保留：

  • 架构决策。
  • 未解决问题。
  • 关键文件和实现细节。
  • 当前任务状态。

结构化笔记

智能体将阶段性结论写入外部存储，如 Markdown 笔记、TODO、项目日志等。后续按需检索，降低上下文占用。

子代理架构

主代理负责规划和综合，子代理在独立上下文中并行探索，最后只返回浓缩结果。适合复杂研究、代码分析等任务。

7. ContextPacket 与 ContextConfig

ContextPacket 是候选信息单元，包含：

  • 内容。
  • 时间戳。
  • token 数。
  • 相关性分数。
  • 元数据。

ContextConfig 管理上下文构建参数：

  • max_tokens：最大 token 数。
  • reserve_ratio：系统指令预留比例。
  • min_relevance：最低相关性阈值。
  • recency_weight：新近性权重。
  • relevance_weight：相关性权重。
  • enable_compression：是否启用压缩。

8. NoteTool：结构化笔记

NoteTool 面向长时程任务，用 Markdown + YAML Front-matter 保存结构化笔记。

适用场景：

  • 项目状态追踪。
  • 阶段性结论记录。
  • 阻塞点管理。
  • 行动项维护。
  • 研究资料整理。

常见笔记类型包括：

  • task_state
  • conclusion
  • blocker
  • action
  • reference

相比向量记忆，NoteTool 更轻量、更适合人类阅读和版本管理。

9. TerminalTool：即时环境检索

TerminalTool 让智能体通过终端操作获取上下文，例如查看目录、读取文件、搜索内容。它体现了 JIT 上下文思想：不把整个项目塞进上下文，而是让 Agent 按需探索。

使用时应注意权限和安全边界，避免危险命令和敏感信息泄露。

10. 本章重点

  • 上下文工程是提示工程的扩展，关注完整输入状态管理。
  • 长上下文不等于高质量上下文，信息筛选更重要。
  • 工具、记忆、RAG、笔记和文件系统都属于上下文来源。
  • GSSC 是可复用的上下文构建流程。
  • 长时程任务需要压缩、笔记和子代理协同。
  • ContextBuilder、NoteTool、TerminalTool 构成 HelloAgents 的上下文工程基础。

11. 小结

第九章把智能体开发从“写提示词”推进到“管理上下文状态”。一个强健的智能体系统，关键不只是模型能力，而是能否在每次调用前为模型提供精炼、相关、结构清晰且可持续维护的上下文。
