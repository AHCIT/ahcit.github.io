第07章 构建你的Agent框架

1. 本章目标

成熟框架虽然功能强，但也存在几个典型问题：

  • 抽象层复杂：概念多、学习曲线陡。
  • 版本变化快：API 频繁变动，维护成本高。
  • 黑盒化严重：核心逻辑封装过深，不利于理解 Agent 原理。
  • 依赖复杂：容易引入依赖冲突和部署负担。

自建框架的价值不在于重复造轮子，而在于通过亲手实现核心组件，真正理解 Agent 的运行机制，并获得完全的定制能力。

2. HelloAgents 的设计理念

HelloAgents 追求的是“轻量但完整”。它的关键设计理念包括：

  1. 轻量级与教学友好：依赖少、代码结构清晰，便于逐章理解。
  2. 基于标准 API：围绕 OpenAI 兼容接口构建，方便接入不同模型服务。
  3. 渐进式学习：每章迭代一个可安装版本，逐步增加能力。
  4. 万物皆工具：除核心 Agent 外，Memory、RAG、MCP、RL 等能力都抽象为 Tool，减少额外概念负担。

3. 框架整体架构

核心原则是：

  • 分层解耦：核心、Agent 实现、工具系统分离。
  • 职责单一：每个模块只做一类事情。
  • 接口统一：Agent 和 Tool 都有一致的调用入口。

4. HelloAgentsLLM 扩展

HelloAgentsLLM 是模型调用中枢，本章在前面基础上增强三类能力。

多提供商支持

通过 provider 参数支持 OpenAI、ModelScope、智谱等服务商。不同服务商的 API Key、Base URL 和默认模型由框架内部处理，用户只需切换 provider。

推荐扩展方式是继承 HelloAgentsLLM，重写特定 provider 的初始化逻辑，而不是直接修改框架源码。

本地模型调用

VLLM 和 Ollama 都能提供 OpenAI 兼容接口，因此可以无缝接入：

  • VLLM：适合高吞吐、本地生产级推理。
  • Ollama：适合快速运行本地模型。

只要配置 base_url、api_key 和 model，Agent 代码无需修改。

自动检测机制

自动检测按优先级判断 provider：

  1. 检查特定环境变量，如 MODELSCOPE_API_KEY、OPENAI_API_KEY。
  2. 根据 LLM_BASE_URL 判断云服务或本地服务端口。
  3. 根据 API Key 格式做辅助判断。

这样可以减少显式配置，提升易用性。

5. 核心接口实现

Message

Message 统一框架内部消息格式，包含：

  • role：user、assistant、system、tool。
  • content：消息内容。
  • timestamp：时间戳。
  • metadata：扩展信息。

to_dict() 方法负责转换为 OpenAI API 兼容格式。

Config

Config 集中管理默认模型、温度、最大 token、日志级别、历史长度等配置，并支持从环境变量读取。

Agent 抽象基类

Agent 是所有智能体的统一父类，定义：

  • name
  • llm
  • system_prompt
  • config
  • _history
  • 抽象方法 run()
  • 历史消息管理方法

通过抽象基类约束不同 Agent 的统一接口。

6. Agent 范式框架化

本章将第四章的经典范式重新纳入框架体系。

SimpleAgent

SimpleAgent 是最基础的对话 Agent，支持：

  • 系统提示词注入。
  • 历史消息管理。
  • 可选工具调用。
  • 流式响应。
  • 动态添加和移除工具。

它是后续复杂 Agent 的最小可用版本。

ReActAgent

ReActAgent 采用“Thought + Action”循环：

  1. 构建包含工具说明和历史记录的 Prompt。
  2. LLM 输出 Thought 和 Action。
  3. 若 Action 是工具调用，则执行工具并追加 Observation。
  4. 若 Action 是 Finish，则返回最终答案。
  5. 达到最大步数则终止。

框架化后的重点是接入统一的 ToolRegistry。

ReflectionAgent

ReflectionAgent 引入“生成—反思—改进”循环，用于提高答案质量。它适合需要自我检查和多轮修正的任务。

Plan-and-SolveAgent

Plan-and-SolveAgent 先生成计划，再按步骤执行，适合复杂任务分解。它强调任务拆解能力和执行过程的可控性。

7. 工具系统

工具系统是 HelloAgents 的关键抽象，包含：

  • Tool：所有工具的基类。
  • ToolRegistry：工具注册、查找、执行中心。
  • ToolChain：支持多个工具顺序执行。
  • AsyncToolExecutor：支持异步工具调用。
  • 内置工具：如计算器、搜索工具。

工具统一暴露名称、描述、参数和 run() 方法，Agent 只需要面向统一接口调用。

8. 本章重点

  • HelloAgents 的核心目标是教学友好、轻量、可扩展。
  • 统一 LLM 接口是框架的基础。
  • Message、Config、Agent 构成核心抽象。
  • 不同 Agent 范式通过继承基类实现统一接口。
  • “万物皆工具”降低了系统复杂度。
  • ToolRegistry 是 Agent 与外部能力连接的关键。

9. 小结

第七章完成了 HelloAgents 框架的底座建设：统一模型调用、消息系统、配置系统、Agent 抽象和工具系统。后续的记忆、检索、通信协议、强化学习和应用案例，都会在这个框架基础上继续扩展。
