第04章 智能体经典范式

1. 本章目标

现代智能体的核心能力，是将大语言模型的推理能力与外部世界连接起来，使其能够：

  • 理解用户意图
  • 拆解复杂任务
  • 调用搜索、代码解释器、API 等工具
  • 根据外部反馈动态调整
  • 最终完成目标

本章聚焦三种经典智能体范式：

范式 核心思想 关键词
ReAct 边思考边行动，根据观察动态调整 Thought-Action-Observation
Plan-and-Solve 先完整规划，再逐步执行 Plan → Solve
Reflection 执行后自我反思并优化 Execute → Reflect → Refine

3. ReAct 范式

3.1 核心思想

ReAct = Reasoning + Acting，由 Shunyu Yao 等人在 2022 年提出。

它将推理和行动结合起来，形成循环：

阶段 作用
Thought 分析当前情况、拆解任务、规划下一步
Action 调用工具或输出最终答案
Observation 工具执行后的外部反馈

ReAct 的关键在于：

思考指导行动，行动结果反过来修正思考。

3.2 形式化表达

在时间步 $t$，模型根据问题 $q$ 和历史行动-观察轨迹生成当前思考和行动：

$$
(th_t,a_t)=\pi(q,(a_1,o_1),\ldots,(a_{t-1},o_{t-1}))
$$

工具执行行动并返回观察：

$$
o_t=T(a_t)
$$

3.3 适用场景

  • 查询实时信息：天气、新闻、股价、产品信息
  • 精确计算：调用计算器避免 LLM 算错
  • 与 API 交互：数据库查询、服务调用、业务操作

3.4 工具设计

工具是智能体连接外部世界的“手和脚”。一个工具通常包含三部分：

要素 说明
Name 工具唯一标识，如 Search
Description 工具用途说明，供 LLM 判断何时使用
Execution Logic 真正执行任务的函数

本章示例工具：基于 SerpApi 的网页搜索工具。

工具执行器 ToolExecutor 负责：

  • 注册工具：registerTool(name, description, func)
  • 获取工具：getTool(name)
  • 输出工具描述：getAvailableTools()

3.6 ReAct 特点

优点：

  • 可解释性强：Thought 展示完整决策轨迹
  • 动态纠错：可根据 Observation 调整下一步
  • 工具协同：结合 LLM 推理和外部工具能力
  • 适合开放探索任务：尤其是信息不确定、需要边查边判断的场景

局限：

  • 强依赖 LLM 的推理、格式遵循和工具选择能力
  • 多轮调用导致延迟和成本较高
  • Prompt 脆弱，模型可能不按格式输出
  • 步进式决策可能缺乏全局规划，陷入局部最优或循环

调试技巧：

  • 打印完整 Prompt
  • 查看 LLM 原始输出
  • 检查工具输入输出格式
  • 加入 Few-shot 示例
  • 使用更强模型或将 temperature 设为 0

4. Plan-and-Solve 范式

4.1 核心思想

Plan-and-Solve = 先规划，后执行。

与 ReAct “走一步看一步”不同，Plan-and-Solve 更像建筑师先画蓝图，再按图施工。

流程：

4.2 工作原理

阶段 作用
Planning Phase 将复杂问题分解成清晰、有序、可执行的步骤
Solving Phase 严格按照计划逐步执行，每步利用历史结果

形式化表达：

规划模型生成计划：

$$
P=\pi_{plan}(q)
$$

其中：

$$
P=(p_1,p_2,\dots,p_n)
$$

执行模型逐步完成第 $i$ 个步骤：

$$
s_i=\pi_{solve}(q,P,(s_1,\dots,s_{i-1}))
$$

最终答案为最后一步结果 $s_n$。

4.3 适用场景

  • 多步数学应用题
  • 报告撰写与资料整合
  • 代码生成前的模块规划
  • 结构清晰、逻辑路径较确定的复杂任务

4.6 特点与局限

优点：

  • 全局结构清晰
  • 适合长链路、多步骤任务
  • 避免中途偏离目标
  • 每一步状态明确，便于追踪

局限：

  • 计划通常是静态的，执行中若发现计划不合理，需要额外重规划机制
  • 不适合高度动态、需要频繁外部反馈的任务
  • 规划阶段错误会影响后续所有步骤

5. Reflection 范式

5.1 核心思想

Reflection 为智能体引入“事后自我校正循环”。

流程：

阶段 作用
Execution 生成初始方案或答案
Reflection 审视初稿，指出错误和改进方向
Refinement 根据反馈生成优化版本

它模拟人类的校对、验算、代码评审过程。

5.2 形式化表达

第 $i$ 次输出为 $O_i$，反思模型生成反馈：

$$
F_i=\pi_{reflect}(Task,O_i)
$$

优化模型根据任务、上一版输出和反馈生成新版输出：

$$
O_{i+1}=\pi_{refine}(Task,O_i,F_i)
$$

5.3 Reflection 的价值

  • 提供内部纠错回路
  • 能修正逻辑错误、事实错误和策略问题
  • 将一次性执行转为持续优化
  • 通过轨迹记录形成短期记忆
  • 适合高质量、高可靠性任务

核心循环：

终止条件：

  • 反馈中包含“无需改进”
  • 达到最大迭代次数

5.7 成本收益分析

成本：

  • 模型调用次数增加
  • 任务延迟提高
  • Prompt 设计更复杂

收益：

  • 结果质量显著提升
  • 可靠性和鲁棒性增强
  • 能从“可用方案”迭代到“高质量方案”

适合场景：

  • 关键业务代码生成
  • 技术报告和研究报告
  • 复杂逻辑推演
  • 决策支持系统

不适合场景：

  • 强实时任务
  • 对质量要求不高的快速响应任务

6. 三种范式对比

维度 ReAct Plan-and-Solve Reflection
核心流程 思考-行动-观察循环 先规划后执行 执行-反思-优化
决策方式 动态步进 静态计划为主 迭代改进
是否依赖工具 通常依赖 可依赖，也可纯推理 不一定依赖
优势 灵活、可解释、适应环境 结构清晰、目标稳定 质量高、可靠性强
局限 成本高，可能循环 计划错误会传导 延迟高，调用成本高
适合任务 搜索、API 操作、开放探索 数学题、报告、结构化任务 代码优化、审稿、复杂决策

选择建议：

任务特点 推荐范式
需要实时外部信息 ReAct
任务结构清晰、步骤明确 Plan-and-Solve
对质量和可靠性要求极高 Reflection
复杂真实业务 混合使用三者

7. 混合范式思路

实际智能体系统中，三种范式可以组合使用：

例如客服退款智能体：

  1. Plan-and-Solve：规划处理流程
  2. ReAct：查询订单、物流、政策工具
  3. Reflection：对争议决策进行复核
  4. 最终生成回复邮件

8. 本章要点总结

  1. 智能体经典范式的核心，是组织 LLM 的“思考”和“行动”过程
  2. ReAct 通过 Thought-Action-Observation 循环实现动态决策和工具调用
  3. ReAct 适合需要外部信息、实时反馈和工具协作的任务
  4. Plan-and-Solve 将任务拆为规划和执行两阶段，适合结构化、多步骤任务
  5. Plan-and-Solve 的关键是计划质量和执行时的状态管理
  6. Reflection 通过执行、反思、优化循环提升最终结果质量
  7. Reflection 本质上是以更多模型调用成本换取更高可靠性
  8. 三种范式没有绝对优劣，应根据任务特征选择或组合使用
  9. 亲手实现这些范式，有助于理解智能体框架背后的设计机制
