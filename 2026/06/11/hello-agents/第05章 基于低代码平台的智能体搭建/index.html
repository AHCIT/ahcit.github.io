第05章 基于低代码平台的智能体搭建

1. 平台化构建的意义

第四章通过代码实现了 ReAct、Plan-and-Solve、Reflection 等经典智能体范式，但在真实项目中，纯代码开发并不总是最高效。低代码平台的价值在于把底层复杂能力封装为可视化模块，让开发者更专注于业务逻辑、工作流设计和提示词优化。

1.1 为什么需要低代码平台

价值 说明
降低技术门槛 将 API 调用、状态管理、工具调用等封装为节点，非技术人员也能参与构建
提升开发效率 适合快速验证原型，将精力放在业务逻辑和 Prompt 优化上
可视化与可观测性 能直观看到数据流、节点耗时、工具调用失败点
标准化最佳实践 内置 ReAct 模板、RAG 检索、工具规范等能力

低代码不是取代代码，而是提供更高层次抽象，使智能体开发从“底层实现”转向“业务编排”。

2. 低代码平台概览

本章重点介绍四个平台：Coze、Dify、FastGPT、n8n。

平台 核心定位 适合人群 典型场景
Coze 零代码/低代码 Agent 构建平台 产品、运营、个人创作者 快速搭建 Bot、多渠道发布
Dify 开源 LLM 应用开发与运营平台 开发者、企业团队 企业级 Agent、RAG、工作流、多模态
FastGPT 开源知识库问答与 Agent 平台 中小企业、RAG 开发者 私有知识库问答、智能客服
n8n 通用工作流自动化平台 需要业务系统集成的团队 邮件、CRM、数据库、API 自动化

3. Coze：快速搭建与分发智能体

3.1 平台特点

Coze 以友好的可视化体验和丰富插件生态见长，适合快速构建、测试和发布智能体。

核心模块可类比为：

Coze 模块 类比 作用
工作流 通关路线图 编排任务执行流程
对话流 NPC 对话路线 设计多轮对话逻辑
插件 技能卡 扩展外部能力
知识库 百科全书 提供私有知识支撑
提示词 控制键 约束智能体行为
数据库 云存档 存储业务数据
发布管理 关卡审核员 多渠道发布
模型管理 角色库 选择底层模型
效果评测 闯关评分 评估智能体表现

3.2 案例：每日 AI 简报助手

目标：构建一个能够从多源信息中自动聚合 AI 新闻、论文和开源项目的简报智能体。

核心能力

  • 多源信息聚合：RSS、GitHub、arXiv
  • 智能筛选：过滤非 AI 内容、广告、营销信息
  • 结构化输出：技术新闻、学术论文、开源项目
  • 多渠道发布：微信、飞书、豆包等

信息源配置

信息源 插件 用途
36氪、虎嗅、IT之家、InfoQ RSS 获取 AI 技术新闻
GitHub GitHub 插件 获取热门 AI 开源项目
arXiv arXiv 插件 获取 AI 学术论文

Prompt 设计要点

系统提示负责定义长期行为：

  • 角色：资深科技媒体编辑
  • 输出格式：AI 日报 + 日期 + 作者
  • 内容范围：AI、LLM、AIGC、大模型
  • 每条内容必须包含标题、链接、概述
  • 每条新闻/论文/项目添加独立 Emoji

用户提示负责定义具体任务：

  • 从 RSS 中筛选 10 条 AI 技术新闻
  • 从 arXiv 中整理 5 篇 AI 学术论文
  • 从 GitHub 中筛选 5 个 AI 开源项目

3.3 Coze 优势与局限

维度 内容
优势 插件生态丰富、可视化编排直观、提示词控制灵活、多渠道发布便捷
局限 不支持 MCP、部分插件配置复杂、标准化导入导出能力不足

适合场景：

  • 快速验证创意
  • 面向 C 端的轻量智能体
  • 非技术人员参与构建
  • 需要快速发布到多个平台

4. Dify：企业级 LLM 应用开发平台

4.1 平台定位

Dify 是开源 LLM 应用开发平台，融合 BaaS 和 LLMOps，提供从原型设计到生产部署的完整能力。

核心特点：

  • 模型中立，支持 GPT、DeepSeek、Llama、兼容 OpenAI API 的模型
  • 支持本地部署和云端部署
  • 内置 Agent 工作流、RAG Pipeline、数据标注、微调等能力
  • Marketplace 插件生态丰富

4.2 Dify 插件生态

Marketplace 包含：

  • Models：模型插件
  • Tools：工具插件
  • Agent Strategies：智能体策略
  • Extensions：扩展能力
  • Bundles：插件组合包

插件生态决定了智能体能否快速接入外部能力，是平台竞争力的重要组成。

4.6 Dify 优势与局限

维度 内容
优势 全栈式开发体验、低代码与高扩展性平衡、企业级安全、插件生态丰富、开源社区活跃

5. FastGPT：知识库问答与 RAG 平台

5.1 平台定位

FastGPT 是开源知识库问答平台与 Agent 构建工具，核心定位是“企业级 AI 生产力引擎”。

它围绕 RAG 全链路进行优化：

5.2 核心能力

能力 说明
多格式文档导入 支持 Word、Markdown、PDF 等

6. n8n：通用自动化中的 AI Agent

6.1 平台定位

n8n 本质是通用工作流自动化平台，不是纯 LLM 平台。它的核心优势是“连接”：将邮件、数据库、API、SaaS 服务与 LLM 串联起来。

6.2 基本概念

概念 说明
Node 执行具体操作的最小单元
Trigger Node 工作流起点，如邮件到达、定时任务、Webhook
Regular Node 处理数据和逻辑，如调用 LLM、写数据库
Workflow 多个节点连接成的自动化流程

数据在节点之间以 JSON 形式传递。

6.3 案例：智能邮件助手

目标：当收到邮件后，AI Agent 自动理解邮件内容、查询公开信息和私有知识库，并自动生成回复。

整体流程：

6.4 私有知识库构建

准备流程：

  1. Code 节点写入知识文本
  2. Embeddings 节点将文本向量化
  3. Simple Vector Store 存入向量库
  4. 主 Agent 通过相同 Memory Key 检索

示例知识：

  • 工作时间：周一至周五 9:00-17:00
  • 非工作时间无法立即回复
  • 非工作时间收到邮件时应告知对方将在下个工作日处理

6.5 AI Agent 节点配置

AI Agent 节点整合：

组件 作用
Chat Model 大模型推理核心
Memory 保存邮件线程上下文
SerpAPI 搜索公开信息
Simple Vector Store 查询私有知识库
Gmail Send 执行最终回复

Prompt 设计关键：

  • 提取邮件核心问题
  • 优先使用 SerpAPI 查找答案
  • 查询私有知识库获取工作时间
  • 根据当前时间判断回复前缀
  • 最终输出严格 JSON
  • body 内换行使用 <br>

6.6 n8n 优势与局限

维度 内容
优势 连接能力强、可视化自动化、AI Agent 节点集成模型/记忆/工具、支持私有化部署
局限 复杂工作流调试较繁琐、Simple Memory/Vector Store 非持久化、版本控制和协作不如代码成熟、高并发性能有限

适合场景：

  • 邮件自动化
  • CRM/ERP/OA 系统联动
  • 业务流程自动化
  • 将 AI 嵌入已有工作流

7. 四个平台选型建议

需求 推荐平台
快速原型验证、非技术用户、C 端 Bot Coze
企业级应用、复杂业务逻辑、多模态与插件生态 Dify
私有知识库问答、智能客服、RAG 全流程控制 FastGPT
深度业务系统集成、通用自动化流程 n8n

开发模式选择：

模式 适用场景
低代码 快速验证、标准流程、团队技术能力有限
纯代码 高性能、高并发、强定制、复杂底层逻辑
混合开发 平台处理通用流程，代码处理特殊逻辑

8. 本章要点总结

  1. 低代码平台通过可视化和模块化降低智能体开发门槛
  2. 低代码适合快速原型验证，但不等于完全替代代码开发
  3. Coze 强在零代码体验、插件生态和多渠道发布
  4. Dify 强在企业级 LLM 应用开发、插件生态和本地部署
  5. FastGPT 强在 RAG 知识库问答和 MCP 工具接入
  6. n8n 强在连接业务系统和构建端到端自动化流程
  7. Prompt、工具、知识库和工作流编排是低代码智能体构建的核心
  8. 平台选型应围绕业务场景、数据安全、扩展性、成本和团队能力综合判断
  9. 最佳实践通常是低代码 + 代码的混合开发模式
